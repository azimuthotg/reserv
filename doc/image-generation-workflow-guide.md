# คู่มือคำสั่งงานร่วมกันสำหรับการผลิตป้ายและเจนภาพ

<!-- Codex edit: source document for DOCX and infographic generation -->

งานนี้ใช้เป็นคู่มือกลางสำหรับคุยงานป้ายประชาสัมพันธ์ ระบบจองพื้นที่บริการ Smart Creative Learning Space

สำนักวิทยบริการ มหาวิทยาลัยนครพนม

## หลักคิด

เราจะไม่กระโดดจากไอเดียไปเป็นภาพ final ทันที เพราะจะกินเวลาและทรัพยากรสูงเกินจำเป็น
ทุกงานควรเดินผ่านลำดับนี้:

1. คุยขอบเขตงาน
2. ตรวจองค์ประกอบ
3. ทำ brief
4. ทำร่างคุณภาพต่ำหรือ wireframe
5. ตรวจข้อความและ layout
6. ค่อย render ภาพคุณภาพสูง
7. export ไฟล์ใช้งานจริง
8. บันทึกงานและปิดงาน

## คำสั่งหลักที่ใช้ร่วมกัน

| คำสั่ง | ใช้เมื่อ | ผลลัพธ์ที่ต้องได้ |
|---|---|---|
| `START JOB: ...` | เปิดงานใหม่ | สรุปขอบเขต อ่าน context และเช็กสถานะ git |
| `CHECK ASSETS: ...` | ตรวจไฟล์ประกอบ | รู้ว่าไฟล์ไหนใช้เป็นฉาก ตัวละคร โลโก้ QR หรือ reference |
| `MAKE BRIEF: ...` | ก่อนเจน/ออกแบบ | ได้ brief ที่มีเป้าหมาย กลุ่มเป้าหมาย mood/tone และข้อห้าม |
| `DRAFT LOW: ...` | ต้องการร่างเร็ว | ได้ภาพร่างหรือ wireframe คุณภาพต่ำเพื่อดูโครง |
| `LAYOUT DRAFT: ...` | ต้องดูป้ายรวมข้อความจริง | ได้ layout ที่มีข้อความ โลโก้ และ QR จริง |
| `REVIEW DRAFT: ...` | ตรวจร่างก่อนแก้ | ได้รายการจุดดี จุดเสี่ยง และสิ่งที่ควรปรับ |
| `FINAL RENDER: ...` | หลังยืนยัน layout | ได้ภาพประกอบคุณภาพสูงตาม brief ที่อนุมัติแล้ว |
| `EXPORT: ...` | เตรียมส่งใช้งาน | ได้ไฟล์ print/web และตรวจ QR จากไฟล์จริง |
| `RECORD JOB` | หลังจบรอบงาน | อัปเดตเอกสาร progress พร้อมไฟล์/commit อ้างอิง |
| `END JOB` | ปิดงาน | สรุปสิ่งที่เสร็จ ไฟล์ที่ได้ และสิ่งที่ยังค้าง |

## กติกาสำคัญ

- ห้ามข้ามจาก brief ไป final render ถ้ายังไม่ได้ตรวจ `DRAFT LOW` หรือ `LAYOUT DRAFT`
- ห้ามให้ AI สร้าง QR Code หรือโลโก้เอง แต่ต้องนำไฟล์จริงที่เตรียมไว้ไปประกอบใน layout
- ข้อความสำคัญต้องวางด้วย layout tool หรือ HTML/CSS ไม่ฝังในภาพ AI
- Codex ต้องช่วยวางข้อความจริง โลโก้จริง และ QR จริงให้จบเป็นไฟล์พร้อมตรวจ ไม่ใช่ปล่อยให้ไปทำต่อในโปรแกรมอื่น
- ภาพ AI ใช้เป็นภาพประกอบ ไม่ใช่แหล่งความจริงของข้อความหรือ QR
- ถ้าต้องการตรวจตัวหนังสือ ให้ใช้ `LAYOUT DRAFT` ไม่ใช่ `FINAL RENDER`
- ถ้าต้องการแค่ดูโครง ให้ใช้ภาพคุณภาพต่ำก่อน
- ทุกไฟล์ที่ใช้จริงต้องเก็บ path ชัดเจนใน workspace

## ไฟล์จริงที่ต้องนำมาประกอบในงานป้าย

ไฟล์เหล่านี้ไม่ให้ AI วาดหรือสร้างขึ้นใหม่ แต่ต้องนำมาวางในขั้น `LAYOUT DRAFT` และ `EXPORT`:

| ประเภท | ไฟล์ |
|---|---|
| โลโก้มหาวิทยาลัยนครพนม | `doc/illustrations/logo_NPU.png` |
| โลโก้สำนักวิทยบริการ | `doc/illustrations/logo_ARC.png` |
| QR Code LINE OA | `doc/illustrations/QR_LINEOA_ARC.jpg` |
| QR Code ระบบจอง | `doc/illustrations/QR_WebApp_ARC.png` |

ข้อความจริงบนป้ายให้ Codex ช่วยจัดวางในไฟล์ layout โดยตรง เพื่อให้ตรวจคำไทย ขนาดตัวอักษร ลำดับสายตา และการสแกน QR ได้ในรอบเดียว

## Workflow แนะนำสำหรับงานป้าย

### 1. START JOB

ใช้เปิดงานและตั้งขอบเขต เช่น:

```text
START JOB: ป้ายที่ 1 ป้ายเปิดตัวระบบจอง
```

Codex ต้องอ่าน context, progress, poster plan และสถานะ git ก่อนตอบแผนงาน

### 2. CHECK ASSETS

ใช้ตรวจวัตถุดิบ เช่น:

```text
CHECK ASSETS: ป้ายที่ 1
```

สิ่งที่ต้องตรวจ:

- ภาพฉากจริงของห้องสมุด
- ภาพบุคลากร reference
- ภาพตัวละคร support เช่น `personal_akeky.png`
- โลโก้จริง `logo_NPU.png` และ `logo_ARC.png`
- QR จริง `QR_LINEOA_ARC.jpg` และ `QR_WebApp_ARC.png`
- mood/tone reference เช่น `example1.jpg`

### 3. MAKE BRIEF

ใช้สรุปก่อนสร้างภาพ เช่น:

```text
MAKE BRIEF: ป้ายเปิดตัวระบบจอง
```

brief ต้องตอบให้ครบ:

- ป้ายนี้บอกอะไร
- ใครคือกลุ่มเป้าหมาย
- ข้อความหลักคืออะไร
- ภาพหลักคืออะไร
- mood/tone เป็นแบบไหน
- อะไรห้ามเจน เช่น QR, โลโก้, ตัวอักษรสำคัญ

### 4. DRAFT LOW

ใช้ทำภาพร่างเร็ว เช่น:

```text
DRAFT LOW: ป้ายที่ 1 โครง A3 แนวตั้ง
```

เป้าหมาย:

- เห็นตำแหน่งตัวละคร
- เห็นพื้นที่หัวข้อ
- เห็นพื้นที่ QR
- เห็นลำดับสายตา
- ยังไม่ต้องคม ไม่ต้องละเอียด

### 5. LAYOUT DRAFT

ใช้ประกอบป้ายแบบมีข้อความจริง เช่น:

```text
LAYOUT DRAFT: ป้ายที่ 1 ใส่ข้อความจริงและ QR จริง
```

สิ่งที่ต้องตรวจ:

- ไฟล์จริงถูกนำมาวางครบหรือไม่: `logo_NPU.png`, `logo_ARC.png`, `QR_LINEOA_ARC.jpg`, `QR_WebApp_ARC.png`
- หัวข้อใหญ่พอหรือไม่
- QR เด่นและสแกนง่ายหรือไม่
- โลโก้ไม่เล็กเกินไป
- ข้อความไม่แน่นเกินไป
- `personal_akeky.png` เป็นมุม support ไม่แย่งความเด่น

### 6. REVIEW DRAFT

ใช้ให้ตรวจร่าง เช่น:

```text
REVIEW DRAFT: ป้ายที่ 1
```

ผลลัพธ์ควรมี:

- จุดที่ดี
- จุดที่ควรแก้
- ความเสี่ยงด้านการอ่านและการสแกน QR
- ข้อเสนอสำหรับรอบถัดไป

### 7. FINAL RENDER

ใช้เมื่อ layout ผ่านแล้วเท่านั้น เช่น:

```text
FINAL RENDER: ภาพประกอบป้ายที่ 1
```

Codex ต้องไม่เปลี่ยนโครงหลักโดยพลการ และต้องใช้ brief ที่อนุมัติแล้ว

### 8. EXPORT

ใช้สร้างไฟล์ส่งใช้งาน เช่น:

```text
EXPORT: ป้ายที่ 1 A3 print และ PNG web
```

ไฟล์ที่ควรได้:

- ไฟล์ print ความละเอียดสูง
- ไฟล์ web/social
- ไฟล์ source ที่แก้ไขได้
- รายงานว่า QR ถูกตรวจแล้วจากไฟล์จริง
- ไฟล์ final ที่ประกอบข้อความจริง โลโก้จริง และ QR จริงครบแล้ว

### 9. RECORD JOB

ใช้บันทึกงาน เช่น:

```text
RECORD JOB: บันทึก brief และ prompt ป้ายที่ 1
```

ต้องอัปเดตเอกสารที่เกี่ยวข้อง เช่น:

- `doc/poster-plan.md`
- `doc/progress-YYYY-MM-DD.md`
- path ของ asset ที่สร้าง
- commit hash ถ้ามีการ commit/push

### 10. END JOB

ใช้ปิดงาน เช่น:

```text
END JOB: ป้ายที่ 1 รอบ draft
```

ต้องสรุป:

- เสร็จอะไรแล้ว
- ไฟล์อยู่ที่ไหน
- commit ล่าสุดคืออะไร
- งานที่ยังค้างคืออะไร

## บทสวดใช้งานจริงในช่องแชท

หัวข้อนี้คือคู่มือแบบ copy/paste สำหรับทีมงาน ใช้สั่งงานในช่องแชททีละรอบ โดยไม่ต้องเดาเองว่าต้องเริ่มยังไง ตรวจอะไร และควรรอผลลัพธ์แบบไหน

### บทสวดรอบที่ 1 — เปิดงาน

พิมพ์ในช่องแชท:

```text
START JOB: ป้ายที่ 1 ป้ายเปิดตัวระบบจอง
```

สิ่งที่ต้องรอจาก Codex:

- สรุปว่าป้ายนี้คือป้ายอะไร
- อ่าน `doc/poster-plan.md`
- อ่าน progress ล่าสุดที่เกี่ยวข้อง
- เช็ก asset ที่มีใน `doc/illustrations`
- เช็กสถานะ git
- บอกขอบเขตงานรอบนี้
- ยืนยันว่าจะยังไม่ final render จนกว่า brief/layout จะผ่าน

สิ่งที่ทีมต้องตรวจ:

- Codex เข้าใจป้ายถูกใบหรือไม่
- ขอบเขตงานไม่กว้างเกินไปหรือไม่
- มีการพูดถึงไฟล์จริงและ workflow ถูกต้องหรือไม่

ถ้าผ่าน ให้สั่งรอบถัดไป:

```text
CHECK ASSETS: ป้ายที่ 1 ใช้ไฟล์ใน doc/illustrations
```

### บทสวดรอบที่ 2 — ตรวจองค์ประกอบ

พิมพ์ในช่องแชท:

```text
CHECK ASSETS: ป้ายที่ 1 ใช้ไฟล์ใน doc/illustrations
```

สิ่งที่ต้องรอจาก Codex:

- รายการไฟล์ทั้งหมดที่เกี่ยวข้อง
- แยกว่าไฟล์ไหนเป็นฉากจริงของห้องสมุด
- แยกว่าไฟล์ไหนเป็นตัวละครหลัก ตัวละครรอง หรือ reference
- แยกว่าไฟล์ไหนเป็นโลโก้จริง
- แยกว่าไฟล์ไหนเป็น QR Code จริง
- บอกว่า `example1.jpg` ใช้เป็น mood/tone เท่านั้น ไม่ลอก layout
- บอกว่า `personal_akeky.png` ใช้เป็นมุม support ขนาดเล็ก
- เตือนว่า AI ห้ามสร้าง QR/โลโก้เอง แต่ layout ต้องนำไฟล์จริงมาใส่

สิ่งที่ทีมต้องตรวจ:

- ไฟล์ครบหรือไม่
- QR LINE OA และ QR ระบบจองเป็นไฟล์ล่าสุดหรือไม่
- โลโก้ถูกต้องหรือไม่
- ภาพบุคลากรได้รับอนุญาตให้ใช้หรือไม่
- มีไฟล์ใดไม่ควรใช้หรือไม่

ถ้าผ่าน ให้สั่งรอบถัดไป:

```text
MAKE BRIEF: ป้ายที่ 1 ใช้โทน ARC/NPU ใส่ QR จริง 2 ช่องทาง และให้ personal_akeky เป็นมุม support
```

### บทสวดรอบที่ 3 — ทำ brief

พิมพ์ในช่องแชท:

```text
MAKE BRIEF: ป้ายที่ 1 ใช้โทน ARC/NPU ใส่ QR จริง 2 ช่องทาง และให้ personal_akeky เป็นมุม support
```

สิ่งที่ต้องรอจาก Codex:

- เป้าหมายของป้าย
- กลุ่มเป้าหมาย
- ข้อความหลัก
- ข้อความรอง
- call to action
- mood/tone
- ภาพหลักที่ควรใช้
- ตัวละครหลักและตัวละครรอง
- ตำแหน่ง QR โดยประมาณ
- รายการไฟล์จริงที่จะนำมาใส่ใน layout
- ข้อห้าม เช่น ห้าม AI สร้าง QR/โลโก้เอง ห้ามฝังข้อความไทยยาวในภาพ AI

สิ่งที่ทีมต้องตรวจ:

- ข้อความถูกต้องหรือไม่
- สั้นพอสำหรับป้ายหรือไม่
- QR LINE OA เป็นช่องทางหลักหรือไม่
- QR เว็บไซต์เป็นช่องทางรองชัดไหม
- `personal_akeky.png` ไม่เด่นเกินไปหรือไม่
- brief สอดคล้องกับภาพลักษณ์สำนักวิทยบริการหรือไม่

ถ้าผ่าน ให้ตอบ:

```text
อนุมัติ brief
```

แล้วสั่งรอบถัดไป:

```text
DRAFT LOW: ป้ายที่ 1 A3 แนวตั้ง เห็นตำแหน่งหัวข้อ โลโก้ QR และตัวละคร
```

### บทสวดรอบที่ 4 — ทำร่างเร็วคุณภาพต่ำ

พิมพ์ในช่องแชท:

```text
DRAFT LOW: ป้ายที่ 1 A3 แนวตั้ง เห็นตำแหน่งหัวข้อ โลโก้ QR และตัวละคร
```

สิ่งที่ต้องรอจาก Codex:

- ภาพร่างหรือ wireframe คุณภาพต่ำ
- เห็นตำแหน่งหัวข้อ
- เห็นตำแหน่งโลโก้
- เห็นตำแหน่ง QR LINE OA และ QR เว็บไซต์
- เห็นตำแหน่งตัวละครหลัก
- เห็นตำแหน่ง `personal_akeky.png` แบบ support
- สรุปเหตุผลของ layout
- บอกชัดว่ายังไม่ใช่ final render

สิ่งที่ทีมต้องตรวจ:

- มองแวบแรกแล้วรู้ไหมว่าเป็นระบบจองของสำนักวิทยบริการ
- หัวข้อเด่นพอหรือไม่
- QR เด่นพอหรือไม่
- ลำดับสายตาเริ่มจากหัวข้อไป QR ได้ไหม
- พื้นที่ QR โล่งพอหรือไม่
- ตัวละครช่วยสื่อสารหรือแย่งความเด่น
- layout แน่นเกินไปหรือไม่

ถ้าต้องแก้ ให้สั่งแบบเจาะจง เช่น:

```text
REVIEW DRAFT: QR LINE OA ยังเล็กไป ขอให้เด่นกว่า QR เว็บไซต์ และลดขนาด personal_akeky ลง
```

ถ้าผ่าน ให้สั่งรอบถัดไป:

```text
LAYOUT DRAFT: ป้ายที่ 1 ใส่ข้อความจริง โลโก้จริง QR จริง และ export PNG สำหรับตรวจ
```

### บทสวดรอบที่ 5 — ทำ layout draft พร้อมข้อความจริง

พิมพ์ในช่องแชท:

```text
LAYOUT DRAFT: ป้ายที่ 1 ใส่ข้อความจริง โลโก้จริง QR จริง และ export PNG สำหรับตรวจ
```

สิ่งที่ต้องรอจาก Codex:

- ไฟล์ PNG หรือ mockup ที่มี layout จริง
- ใช้ข้อความไทยจริง
- ใช้ `doc/illustrations/logo_NPU.png`
- ใช้ `doc/illustrations/logo_ARC.png`
- ใช้ `doc/illustrations/QR_LINEOA_ARC.jpg`
- ใช้ `doc/illustrations/QR_WebApp_ARC.png`
- มี path ของไฟล์ที่สร้าง
- มีรายงานว่าไฟล์ไหนใช้ประกอบบ้าง
- ยังไม่ถือเป็น final print

สิ่งที่ทีมต้องตรวจ:

- คำไทยถูกต้องหรือไม่
- ขนาดตัวหนังสืออ่านง่ายหรือไม่
- QR สแกนจากภาพ draft ได้หรือไม่
- QR LINE OA และ QR ระบบจองไปถูกปลายทางหรือไม่
- โลโก้ไม่แตก ไม่ยืด ไม่ผิดสัดส่วน

## คัมภีร์การเจนภาพสำหรับงานนี้

### ให้ AI ทำ

- สร้างภาพประกอบฉากห้องสมุด
- สร้างตัวละครการ์ตูนจาก reference
- สร้าง mood/tone และบรรยากาศ
- สร้างภาพฐานสำหรับนำไปประกอบป้าย

### ไม่ให้ AI สร้างเอง

- สร้าง QR Code ปลอมหรือ QR จำลอง
- สร้างโลโก้ปลอมหรือวาดโลโก้ใหม่
- สะกดข้อความไทยยาว ๆ ลงในภาพ AI
- ตัดสินข้อมูลนโยบายเอง
- เปลี่ยน layout หลังได้รับอนุมัติแล้ว

แต่ในขั้น `LAYOUT DRAFT` และ `EXPORT` ต้องนำ QR Code จริง โลโก้จริง และข้อความจริงมาประกอบให้เสร็จ

### Prompt ที่ควรมี

- ใช้งานเพื่ออะไร
- ขนาดและแนวภาพ
- ฉากอยู่ที่ไหน
- ตัวละครหลักคือใคร
- ตัวละครรองคือใคร
- mood/tone
- สีหลัก
- พื้นที่ว่างสำหรับข้อความและ QR
- ข้อห้ามชัดเจน
