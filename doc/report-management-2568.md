## บทสรุปผู้บริหาร (Executive Summary)

สำนักวิทยบริการ มหาวิทยาลัยนครพนม ได้ดำเนินโครงการพัฒนาระบบจองพื้นที่บริการ Smart Creative Learning Space ภายใต้กรอบแนวคิด Design Thinking โดยมีจุดมุ่งหมายหลักเพื่อแก้ไขข้อจำกัดของระบบเดิมที่พึ่งพาบริการคลาวด์ภายนอก (Google Apps Script และ Google Sheets) ซึ่งส่งผลให้ระบบมีความล่าช้า ขาดความยืดหยุ่น และมีความเสี่ยงด้านความต่อเนื่องของการให้บริการ

โครงการแบ่งออกเป็น 2 กิจกรรมหลัก โดยกิจกรรมที่ 1 มุ่งพัฒนาระบบจองใหม่บนโครงสร้างพื้นฐานของมหาวิทยาลัย (On-premise) ด้วย Django Framework และฐานข้อมูล MySQL โดยยังรักษาการเชื่อมต่อกับ LINE LIFF เพื่อประสบการณ์ผู้ใช้ที่คุ้นเคย พร้อมเพิ่มช่องทางการจองผ่านเว็บไซต์สำนักฯ กิจกรรมที่ 2 เชื่อมต่อระบบจองกับอุปกรณ์ IoT (Sonoff Smart Plug) ผ่าน Home Assistant เพื่อแจ้งเตือนสถานะอุปกรณ์และส่งการแจ้งเตือนให้ผู้จองผ่าน LINE Messaging API อัตโนมัติ

ปัจจุบัน ทั้ง 2 กิจกรรมดำเนินการเสร็จสมบูรณ์และเปิดใช้งานจริงแล้วที่ `https://lib.npu.ac.th/reserv/` ระบบรองรับผู้ใช้ทั้งกลุ่มนักศึกษาและบุคลากรผ่าน 2 ช่องทาง (LINE OA และเว็บไซต์) โดยไม่มีค่าใช้จ่ายรายเดือนสำหรับบริการคลาวด์ภายนอกอีกต่อไป

**สรุปสถานะกิจกรรม:**

| กิจกรรม | ชื่อ | สถานะ |
|---|---|---|
| กิจกรรมที่ 1 | ปรับปรุงฐานข้อมูลและระบบจอง (Local Booking Core) | ✅ เสร็จสมบูรณ์ — ใช้งานจริงแล้ว |
| กิจกรรมที่ 2 | ระบบแจ้งเตือนและตรวจสอบอุปกรณ์ (Alert-as-a-Service) | ✅ เสร็จสมบูรณ์ — ใช้งานจริงแล้ว |

## 1. ความเป็นมาและปัญหา

### 1.1 บริบทการให้บริการ

สำนักวิทยบริการ มหาวิทยาลัยนครพนม จัดพื้นที่บริการ Smart Creative Learning Space เพื่อให้นักศึกษาและบุคลากรสามารถใช้พื้นที่สำหรับการศึกษา ทำงานกลุ่ม และกิจกรรมเชิงสร้างสรรค์ โดยพื้นที่บริการประกอบด้วยห้องทั้งหมด 5 ประเภท ดังนี้

| ชื่อห้อง | ลักษณะการใช้งาน |
|---|---|
| Mini Theater | ห้องฉายภาพยนตร์ขนาดเล็ก เหมาะสำหรับการนำเสนอและกิจกรรมกลุ่ม |
| Edutainment Zone | พื้นที่สื่อการเรียนรู้เชิงบันเทิง |
| Canva Studio | ห้องออกแบบกราฟิกและสื่อดิจิทัล |
| AI Learning Zone (Chat-GPT Lab) | ห้องทดลองใช้งาน AI และเทคโนโลยีปัญญาประดิษฐ์ |
| Meeting Room F1 | ห้องประชุมชั้น 1 |

ระบบการจองพื้นที่เป็นกลไกสำคัญในการบริหารพื้นที่ให้มีประสิทธิภาพ ป้องกันการจองซ้ำซ้อน และสร้างหลักฐานการใช้งานสำหรับการรายงานและการประเมินผล

### 1.2 ระบบเดิมและข้อจำกัด

ระบบจองพื้นที่เดิมถูกพัฒนาในลักษณะ Quick Solution โดยใช้เครื่องมือที่สามารถเริ่มต้นได้ทันทีโดยไม่ต้องใช้งบประมาณเพิ่มเติม สถาปัตยกรรมของระบบเดิมมีลักษณะดังนี้

**[รูปที่ 1] สถาปัตยกรรมระบบเดิม (ก่อนพัฒนา)**

```
ผู้ใช้ใน LINE OA
    ↓ กด Flex Message Card
    ↓ เปิด LIFF
HTML Form  (arc.npu.ac.th — external hosting)
    ↓ ผู้ใช้กรอกข้อมูลด้วยตนเอง
Google Apps Script (Web App — Google Cloud)
    ↓ บันทึกข้อมูล
Google Sheets (ฐานข้อมูล — Google Cloud)
    ↓ ตรวจสิทธิ์ผ่าน
testdb.php (external server)
    ↓ ดึงข้อมูลผู้ใช้จาก
ฐานข้อมูล LDAP / Active Directory ของมหาวิทยาลัย
```

> *ภาพที่ 1 แสดงให้เห็นว่าทุกกระบวนการหลักของระบบเดิมพึ่งพาบริการภายนอกองค์กรทั้งสิ้น*

ระบบเดิมพบข้อจำกัดสำคัญ 4 ด้าน:

**1. การพึ่งพาบริการภายนอก (Vendor Lock-in)**
- Google Sheets มีข้อจำกัดด้านจำนวน API calls ต่อวัน ทำให้เกิดความล่าช้าในชั่วโมงเร่งด่วน
- Google Apps Script มี execution time limit (สูงสุด 6 นาที/ครั้ง) ซึ่งอาจทำให้การจองค้างกลางคัน
- หากนโยบายของ Google เปลี่ยนแปลง ระบบอาจหยุดทำงานโดยไม่มีการแจ้งเตือนล่วงหน้า
- **LINE Notify ซึ่งใช้ส่งการแจ้งเตือน ประกาศปิดบริการถาวรตั้งแต่เดือนเมษายน 2568** ทำให้ระบบแจ้งเตือนหยุดทำงานโดยสมบูรณ์

**2. ความปลอดภัยของข้อมูล**
- ข้อมูลการจองและข้อมูลผู้ใช้ทั้งหมดถูกจัดเก็บใน Google Sheets บนระบบคลาวด์ภายนอกองค์กร
- ไม่สอดคล้องกับนโยบายการจัดเก็บข้อมูลภายในองค์กรและแนวทางการรักษาความปลอดภัยข้อมูลส่วนบุคคล
- ขาดกลไก audit trail ที่เป็นมาตรฐาน ไม่สามารถตรวจสอบย้อนหลังได้

**3. ข้อจำกัดด้านฟังก์ชันการทำงาน**
- ไม่สามารถตรวจสอบ conflict การจองพร้อมกันได้อย่างน่าเชื่อถือ (race condition) ทำให้อาจเกิดการจองซ้ำซ้อนได้
- ผู้ใช้ต้องกรอกชื่อ-นามสกุล คณะ สาขา ด้วยตนเองทุกครั้ง เพิ่มโอกาสข้อมูลผิดพลาด
- ไม่สามารถกำหนดวันหยุดหรือประกาศปิดบริการชั่วคราวล่วงหน้าได้
- ไม่มีปฏิทินรวมสำหรับดูการจองทั้งหมด

**4. ความยากในการบำรุงรักษา**
- โค้ด Google Apps Script ไม่มี version control ทำให้ไม่สามารถ rollback เมื่อเกิดปัญหาได้
- การ debug ทำได้ยากเนื่องจากไม่มี structured log
- การแก้ไขโค้ดต้องอาศัยผู้ที่มีสิทธิ์เข้าถึง Google Account เฉพาะเท่านั้น

## 2. กรอบแนวคิดการพัฒนา — Design Thinking

โครงการนี้นำกระบวนการคิดเชิงออกแบบ (Design Thinking) 5 ขั้นตอนมาเป็นกรอบในการพัฒนา เพื่อให้ระบบใหม่ตอบสนองความต้องการของผู้ใช้จริงได้อย่างตรงจุด

| ขั้นตอน | กิจกรรม | ผลลัพธ์ |
|---|---|---|
| **1. Empathize** | สัมภาษณ์เจ้าหน้าที่และผู้ใช้บริการ, วิเคราะห์ปัญหาที่พบบ่อย | รายการปัญหาจากมุมมองผู้ใช้จริง |
| **2. Define** | จำกัดขอบเขตปัญหาหลัก: ความล่าช้า, ข้อมูลอยู่นอกองค์กร, ขาดการแจ้งเตือน | Problem Statement ที่ชัดเจน |
| **3. Ideate** | ออกแบบสถาปัตยกรรมใหม่, เลือก technology stack, วางแผน 4 กิจกรรม | แผนงานและสถาปัตยกรรมระบบ |
| **4. Prototype** | พัฒนาระบบตาม Phase A–E, ทดสอบแต่ละส่วน | ระบบ prototype บน staging |
| **5. Test** | ทดสอบกับผู้ใช้จริง, ปรับแก้ตามความเห็น, deploy production | ระบบ production ที่ใช้งานได้จริง |

ผลจากกระบวนการ Design Thinking นำไปสู่การกำหนด 2 กิจกรรมหลัก:

- **กิจกรรมที่ 1 — Local Booking Core:** ปรับปรุงฐานข้อมูลและระบบจองให้อยู่ภายในองค์กร
- **กิจกรรมที่ 2 — Alert-as-a-Service:** ระบบแจ้งเตือนและตรวจสอบสถานะอุปกรณ์อัตโนมัติ

## 3. กิจกรรมที่ 1 — ปรับปรุงฐานข้อมูลและระบบจอง (Local Booking Core)

### 3.1 ความต้องการระบบ

จากการวิเคราะห์ระบบเดิมและรับฟังความต้องการของผู้ใช้ กำหนดความต้องการระบบดังนี้

**ความต้องการเชิงฟังก์ชัน (Functional Requirements):**

| รหัส | ความต้องการ | ระดับความสำคัญ |
|---|---|---|
| FR-01 | ผู้ใช้จองห้องผ่าน LINE LIFF ได้โดยไม่ต้องออกจาก LINE | สูงสุด |
| FR-02 | ตรวจสอบสิทธิ์ผ่าน LINE userId เชื่อมกับบัญชีมหาวิทยาลัย | สูงสุด |
| FR-03 | ดึงข้อมูลชื่อ-นามสกุล คณะ สาขา จากระบบมหาวิทยาลัยอัตโนมัติ | สูง |
| FR-04 | ตรวจสอบ conflict การจองแบบ real-time ป้องกันการจองซ้ำซ้อน | สูงสุด |
| FR-05 | บริหารจัดการวันหยุดและการปิดบริการชั่วคราวผ่านหน้า Admin | สูง |
| FR-06 | จำกัดการจองล่วงหน้าไม่เกิน 3 วันทำงาน | สูง |
| FR-07 | แจ้งเตือนจองสำเร็จผ่าน LINE ทันทีหลังจองเสร็จ | สูง |
| FR-08 | แจ้งเตือนผู้จองก่อนถึงเวลาใช้งาน 15 นาที | ปานกลาง |
| FR-09 | แจ้งเตือนก่อนหมดเวลาใช้งาน 10 นาที | ปานกลาง |
| FR-10 | ผู้ดูแลระบบจัดการข้อมูลการจองและผู้ใช้ผ่าน Admin Panel | สูง |

### 3.3 Technology Stack และเหตุผลการเลือก

| ส่วนประกอบ | ที่เลือกใช้ | เหตุผล |
|---|---|---|
| Backend Framework | **Django 4.2 / Python 3.12** | มี ORM, Admin Panel, Migration ในตัว เหมาะสำหรับทีมขนาดเล็ก |
| Database | **MySQL 8.0** | ใช้ infrastructure ที่มีอยู่แล้วภายในองค์กร ไม่ต้องลงทุนเพิ่ม |
| WSGI Server | **Waitress** | รองรับ Windows Server ได้โดยตรง ไม่ต้องพึ่ง Linux/Gunicorn |
| Reverse Proxy | **IIS ARR** | Windows Server มี IIS อยู่แล้ว ไม่ต้องติดตั้งเพิ่ม |
| Static Files | **WhiteNoise** | Serve ผ่าน Waitress ได้โดยไม่ต้องตั้ง IIS แยก |
| Process Manager | **NSSM** | จัดการ Python process เป็น Windows Service ได้ง่าย |
| Frontend | **Django Template + Bootstrap 5.3** | ลด complexity เหมาะกับ LIFF single-page |
| Calendar | **FullCalendar v6** | แสดงปฏิทินการจองในรูปแบบรายวัน/สัปดาห์/เดือน |
| LINE Integration | **LIFF SDK 2.15** | ดึง userId โดยตรงภายใน LINE app |

### 3.4 ช่องทางการจอง 2 ช่องทาง

ระบบใหม่รองรับการจองจาก 2 ช่องทางในระบบเดียวกัน:

**ช่องทางที่ 1 — LINE Official Account (มือถือ)**
1. ผู้ใช้กดปุ่ม "จองห้อง" ใน Rich Menu ของ LINE OA สำนักวิทยบริการ
2. ระบบเปิดหน้าจองผ่าน LINE Front-end Framework (LIFF) โดยดึง userId จาก LINE โดยอัตโนมัติ
3. ระบบตรวจสอบว่าผู้ใช้ผูกบัญชีมหาวิทยาลัยแล้วหรือไม่ หากยังไม่ผูกจะนำไปยังหน้าลงทะเบียน
4. เมื่อผ่านการตรวจสิทธิ์ ระบบดึงชื่อ-คณะ-สาขาจากระบบมหาวิทยาลัยโดยอัตโนมัติ ผู้ใช้เพียงเลือกวัน/เวลาและยืนยันการจอง

**ช่องทางที่ 2 — เว็บไซต์สำนักวิทยบริการ (Desktop/Browser)**
1. ผู้ใช้เปิดเว็บไซต์ `https://lib.npu.ac.th/reserv/` ผ่าน browser
2. ล็อกอินด้วยบัญชี LDAP ของมหาวิทยาลัย (username/password)
3. ระบบแสดงฟอร์มจองพร้อมปฏิทินการจองทั้งหมด
4. ดูประวัติการจองและยกเลิกได้จากหน้าจัดการ

### 3.5 การออกแบบฐานข้อมูล

ระบบใช้ฐานข้อมูล MySQL 5 ตาราง ดังนี้

| ตาราง | บทบาท | ข้อมูลสำคัญ |
|---|---|---|
| `booking_room` | ข้อมูลห้องบริการ | ชื่อห้อง, เวลาเปิด-ปิด, สถานะ, รหัส IoT |
| `booking_lineuser` | Cache ผู้ใช้ที่ผูก LINE กับ LDAP | LINE userId, ชื่อจริง, คณะ, สาขา, เวลา cache |
| `booking_booking` | บันทึกการจองทั้งหมด | ห้อง, ผู้จอง, วันที่, เวลาเริ่ม-สิ้นสุด, สถานะ |
| `booking_bookinglog` | Audit Trail ทุก action | การจอง, action, หมายเหตุ, timestamp |
| `booking_holidaydate` | วันหยุดและปิดบริการ | วันที่, คำอธิบาย, สถานะ |

**กลไก Conflict Check ที่น่าเชื่อถือ:**

ระบบใช้ `SELECT FOR UPDATE` ร่วมกับ `transaction.atomic()` เพื่อป้องกัน race condition กรณีผู้ใช้หลายคนจองช่วงเวลาเดียวกันพร้อมกัน ซึ่งเป็นจุดอ่อนสำคัญของระบบเดิมที่ใช้ Google Sheets

**ระบบ Profile Cache:**

เพื่อลดการเรียก API ไปยัง api.npu.ac.th ระบบบันทึกข้อมูลชื่อ-คณะ-สาขาไว้ใน database และ refresh ทุก 30 วัน เมื่อผู้ใช้จองครั้งแรก ระบบใช้เวลา ~800ms ในการดึงข้อมูล แต่ครั้งต่อไปใช้เพียง ~50ms

**การจัดการวันหยุดและข้อจำกัดการจองล่วงหน้า:**

- ผู้ดูแลระบบสามารถเพิ่มวันหยุดราชการหรือประกาศปิดบริการชั่วคราวได้ผ่าน Admin Panel
- ระบบป้องกันไม่ให้จองในวันหยุด และจำกัดการจองล่วงหน้าไม่เกิน 3 วันทำงาน
- วันหยุดที่กำหนดจะถูก sync ไปยัง date picker ในหน้าจองอัตโนมัติ ผู้ใช้ไม่สามารถเลือกวันหยุดได้

## 4. กิจกรรมที่ 2 — ระบบแจ้งเตือนและตรวจสอบอุปกรณ์ (Alert-as-a-Service)

### 4.1 ที่มาและความจำเป็น

เนื่องจาก LINE Notify ปิดบริการถาวรตั้งแต่วันที่ 1 เมษายน 2568 ระบบแจ้งเตือนเดิมจึงหยุดทำงานโดยสมบูรณ์ กิจกรรมนี้จึงพัฒนาระบบแจ้งเตือนใหม่ผ่าน LINE Messaging API (Push Message) ซึ่งเป็นบริการระดับ Official ของ LINE ที่มีความเสถียรและรองรับการพัฒนาต่อในอนาคต พร้อมกันนี้ยังเชื่อมต่อกับอุปกรณ์ IoT ในพื้นที่บริการเพื่อการจัดการอัตโนมัติ

### 4.2 ระบบแจ้งเตือน LINE Messaging API

ระบบส่งการแจ้งเตือนผ่าน LINE อัตโนมัติ **7 กรณี** ครอบคลุมทุกสถานะของการจอง:

| # | กรณี | เวลาส่ง | รูปแบบ/เนื้อหา |
|---|---|---|---|
| 1 | **จองสำเร็จ** | ทันทีหลังจองเสร็จ | Flex Message — สรุปรายละเอียด (ห้อง วันที่ เวลา) + ปุ่ม "ดูการจองของฉัน" |
| 2 | **แจ้งเตือนก่อนใช้งาน 15 นาที** | ก่อนเวลาจอง 15 นาที | Flex Message — เตือนให้เดินทางมา + ปุ่ม "Check-in เลย" |
| 3 | **Check-in สำเร็จ** | ทันทีหลัง Check-in | Flex Message — ยืนยัน Check-in + ปุ่ม "ควบคุมอุปกรณ์ในห้อง" |
| 4 | **ยกเลิกอัตโนมัติ (Auto-cancel)** | เมื่อไม่ Check-in ภายใน 15 นาทีหลังเริ่ม | Text — แจ้งว่าระบบยกเลิกการจองแล้ว เนื่องจากไม่มีการ Check-in |
| 5 | **แจ้งเตือนก่อนหมดเวลา 10 นาที** | ก่อนหมดเวลา 10 นาที | Text — เตือนให้เตรียมเก็บของและออกจากพื้นที่ |
| 6 | **ยกเลิกโดยผู้ใช้** | ทันทีหลังกด "ยกเลิก" | Text — ยืนยันการยกเลิก พร้อมสรุปรายละเอียดที่ถูกยกเลิก |
| 7 | **ยกเลิกโดยเจ้าหน้าที่** | ทันทีที่ Admin ดำเนินการ | Text — แจ้งว่าเจ้าหน้าที่ยกเลิกการจอง พร้อมเหตุผล |

การแจ้งเตือนรายเวลา (กรณีที่ 2, 4, 5) ทำงานผ่าน Windows Task Scheduler ที่รันคำสั่ง `send_reminders` ทุก 1 นาที โดยมีกลไกป้องกันการแจ้งเตือนซ้ำซ้อนด้วย flag `notified_15min`, `notified_10min`, และ `auto_cancelled` ในฐานข้อมูล

### 4.3 การเชื่อมต่ออุปกรณ์ IoT (Sonoff Integration)

ระบบเชื่อมต่อกับ Smart Plug ยี่ห้อ Sonoff ที่ติดตั้งในแต่ละห้องบริการ ผ่านแพลตฟอร์ม Home Assistant (HA) ที่ทำงานบนเครือข่าย Wi-Fi ภายในห้องสมุด โดยอุปกรณ์ทั้งหมด (Django Server, Home Assistant, Sonoff Smart Plug) อยู่ใน LAN เดียวกัน ทำให้ควบคุมได้รวดเร็วและไม่ต้องพึ่งพา internet ภายนอก

ระบบ IoT มี **4 กระบวนการหลัก:**

**กระบวนการที่ 1 — ผู้จองควบคุมอุปกรณ์ด้วยตนเอง**
```
LINE push แจ้งเตือน 15 นาที → ผู้จองกด Check-in
    ↓ POST /api/checkin/ → Django ตรวจสอบ booking ที่ active
    ↓ LINE push "Check-in สำเร็จ" + ปุ่ม "ควบคุมอุปกรณ์ในห้อง"
    ↓ ผู้จองกดปุ่ม → เปิดหน้า /room-control/ (LIFF บนมือถือ)
    ↓ GET /api/room-status/ → Django → HA REST API (GET /api/states/)
    ↓ แสดงสถานะ on/off ของทุกอุปกรณ์ในห้อง
    ↓ ผู้จองกด toggle → POST /api/device-toggle/
    ↓ Django → HA REST API (POST /api/services/switch/toggle)
    ↓ HA → Sonoff Smart Plug (Wi-Fi ภายในห้องสมุด)
```

**กระบวนการที่ 2 — ปิดอุปกรณ์อัตโนมัติเมื่อหมดเวลา**
```
Task Scheduler (ทุก 1 นาที) → send_reminders
    ↓ ตรวจว่า booking.end_time ผ่านแล้ว
    ↓ Django → HA REST API (POST /api/services/switch/turn_off)
    ↓ HA → ปิด Sonoff Smart Plug ทุกตัวในห้อง
    ↓ บันทึก BookingLog: action = 'auto_off'
```

**กระบวนการที่ 3 — รายงานสถานะ IoT ประจำเช้า**
```
Task Scheduler (07:30 ทุกวัน) → morning_iot_report
    ↓ วนทุกห้องและทุกอุปกรณ์ → HA REST API (GET /api/states/)
    ↓ รวบรวมสถานะ: Online / Offline / ไม่ทราบ
    ↓ LINE push สรุปสถานะ → LINE Group ทีมผู้ดูแลระบบ
```

**กระบวนการที่ 4 — เจ้าหน้าที่ตรวจสอบและควบคุม (Staff Portal)**
```
เจ้าหน้าที่เปิด /manage/iot-monitor/ (Desktop)
    ↓ แสดงสถานะอุปกรณ์ทุกห้อง real-time จาก HA REST API
    ↓ สั่ง toggle / turn_on / turn_off → HA → Sonoff (Wi-Fi)
    ↓ แจ้งเตือนสถานะผิดปกติ → LINE Group ทีมผู้ดูแล
```

- **Infrastructure:** Django Server → Home Assistant (Bearer Token) → Sonoff Smart Plug ผ่าน Wi-Fi ภายใน LAN
- **การตั้งค่า:** แต่ละห้องมีรหัส `ha_entity_id` ในฐานข้อมูล Room ทำให้เพิ่มอุปกรณ์ใหม่ได้โดยไม่ต้องแก้โค้ด

ภายหลังจากกิจกรรมที่ 1 และ 2 สำเร็จแล้ว ได้วางแผนการพัฒนาเพิ่มเติมในอนาคต ดังนี้

### กิจกรรมที่ 3 — เสริมความพร้อมด้าน Resilience (Phase 3)

ความพร้อมด้าน Resilience ในบริบทของระบบนี้หมายถึงความสามารถของทั้งระบบและบุคลากรในการดำเนินงานได้อย่างต่อเนื่อง แม้จะมีเหตุผิดปกติหรือสถานการณ์ที่ไม่คาดคิด ระบบควรทนต่อความผิดพลาดโดยไม่ทำให้บริการหยุดชะงัก และเจ้าหน้าที่สามารถจัดการปัญหาเบื้องต้นได้ด้วยตนเองโดยไม่ต้องรอผู้ดูแลระบบ

| ระดับ | ฟีเจอร์ | รายละเอียด | สถานะ |
|---|---|---|---|
| **ระบบ** | Auto-cancel (No-show) | ยกเลิกอัตโนมัติถ้าไม่ Check-in ภายใน 15 นาที ป้องกันห้องว่างโดยไม่มีคนใช้จริง | ✅ พร้อมใช้งาน |
| **ระบบ** | Virtual Card (JsBarcode) | บัตรเข้าใช้บริการดิจิทัล ผู้ใช้แสดงบน มือถือแทนบัตรกระดาษ เชื่อมกับข้อมูล LDAP | ✅ พร้อมใช้งาน |
| **ระบบ** | Walai Integration | ตรวจสอบสิทธิ์การเข้าห้องสมุดผ่านระบบ Walai ก่อนอนุมัติการจอง | ✅ / ⏳ บางส่วน |
| **ระบบ** | Blacklist (No-show) | นับจำนวน no-show สะสม และระงับสิทธิ์ผู้จองที่ไม่มาใช้บริการซ้ำ | ⏳ วางแผน |
| **เจ้าหน้าที่** | Self-service IoT Reset | เมื่อระบบแจ้งเตือนว่าอุปกรณ์ offline เจ้าหน้าที่สามารถเดินไปยังตำแหน่งติดตั้ง Sonoff และกดปุ่ม reset ได้ทันที โดยไม่ต้องรอผู้ดูแลระบบ | ✅ ออกแบบรองรับ |

### กิจกรรมที่ 4 — ขยายพื้นที่และอุปกรณ์ IoT (Phase 4)

- ขยายการเชื่อมต่อ IoT ไปยังห้องบริการเพิ่มเติม
- ติดตั้งอุปกรณ์ Smart Plug เพิ่มเติมตามงบประมาณที่ได้รับ (40,000 บาท)
- Dashboard สถิติการใช้งานสำหรับผู้บริหาร

## 8. สรุป

โครงการพัฒนาระบบควบคุมพื้นที่เรียนรู้อัตโนมัติ Smart Creative Learning Space ในปีงบประมาณ 2568 ประสบความสำเร็จตามเป้าหมายที่กำหนดทั้ง 2 กิจกรรม

**กิจกรรมที่ 1** ดำเนินการย้ายฐานข้อมูลและ logic การทำงานทั้งหมดมาสู่โครงสร้างพื้นฐานภายในมหาวิทยาลัยสำเร็จสมบูรณ์ ระบบใหม่มีความเสถียร ตอบสนองเร็วกว่าระบบเดิม ข้อมูลผู้ใช้และการจองทั้งหมดจัดเก็บภายในองค์กร สอดคล้องกับนโยบายความปลอดภัยของข้อมูล และสามารถขยายฟังก์ชันการทำงานในอนาคตได้อย่างยืดหยุ่น

**กิจกรรมที่ 2** ดำเนินการเชื่อมต่อระบบจองกับอุปกรณ์ IoT และพัฒนาระบบแจ้งเตือนผ่าน LINE Messaging API แทน LINE Notify ที่ปิดบริการแล้ว ช่วยให้ผู้ใช้ได้รับการแจ้งเตือนอัตโนมัติตลอดกระบวนการจอง และเจ้าหน้าที่สามารถติดตามสถานะอุปกรณ์ในพื้นที่บริการได้แบบ real-time

ทั้งสองกิจกรรมเปิดใช้งานจริงที่ `https://lib.npu.ac.th/reserv/` รองรับผู้ใช้ทั้งกลุ่มนักศึกษาและบุคลากรผ่าน 2 ช่องทาง (LINE OA และเว็บไซต์) โดยไม่มีค่าใช้จ่ายบริการ cloud ภายนอก
