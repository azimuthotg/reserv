# Handoff: ระบบบุคคลภายนอกเข้าห้องสมุด → ทีม reserv

**จาก:** โปรเจกต์ api (backend)  **ถึง:** โปรเจกต์ reserv (หน้าเว็บ/ระบบจอง)
**วันที่:** 2026-06-21  **สถานะ backend:** เสร็จ + deploy + verify บน prod แล้ว

---

## 1. ภาพรวม

บุคคลภายนอกที่มาขอเข้าห้องสมุด (ไม่ใช่ นศ./บุคลากร, ไม่มีใน AD) กรอก **เลขบัตรประชาชน 13 หลัก**
→ ระบบออก **รหัส 10 หลัก** (หมุนเวียนรายวัน) → แปลงเป็น **QR** → สแกนที่ประตูเพื่อเข้า

แบ่งงาน 2 ฝั่ง:
- **api (เสร็จแล้ว):** ฐานข้อมูล + ตรวจเลขบัตร + จัดสรรรหัส + endpoint ออกรหัส/เช็กประตู + log
- **reserv (ต้องทำ):** หน้าเว็บกรอกเลขบัตร → เรียก api → **เรนเดอร์ QR** ให้ผู้ใช้

```
[ผู้ใช้] กรอก 13 หลัก ที่หน้าเว็บ reserv
     → reserv backend (มี JWT) เรียก  POST /v2/external/issue/
     → ได้ access_code 10 หลัก
     → reserv เรนเดอร์เป็น QR บนหน้าจอ
[ผู้ใช้] เอา QR ไปสแกนที่ประตู
     → ประตู (มี JWT) เรียก  GET /v2/external/check/<code>/
     → ผ่าน/ไม่ผ่าน
```

---

## 2. สิ่งที่ api เตรียมให้แล้ว (พร้อมใช้บน prod)

Base URL: `https://api.npu.ac.th`

ทุก endpoint เป็น **v2 ต้องแนบ JWT** (`Authorization: Bearer <access_token>`) — เส้นทางเดียวกับที่ระบบอื่นใช้อยู่
ขอ token: `POST /v2/token/` body `{"username","password"}` → ได้ `access` (อายุ 365 วัน)

### 2.1 ออกรหัส (reserv เรียก) — `POST /v2/external/issue/`
```
Request body (JSON):
  { "citizen_id": "1101701579763", "first_name": "สมชาย", "last_name": "ใจดี" }

Response 200:
  { "success": true,
    "access_code": "1084281961",      ← เอาไปทำ QR (สตริง 10 หลักดิบๆ)
    "valid_date": "2026-06-21",       ← ใช้ได้เฉพาะวันนี้ (เวลาไทย)
    "member": { "citizen_id": "...", "first_name": "...", "last_name": "..." } }

Error:
  400  เลขบัตรไม่ถูกต้อง (checksum ไม่ผ่าน) / ไม่ได้ส่งชื่อ-สกุล
  403  สมาชิกถูกระงับ (status revoked)
  503  รหัสในพูลถูกใช้หมดสำหรับวันนี้ (pool เต็ม — ปกติไม่เกิด, pool 100)
```
หมายเหตุ: **คนเดิมขอซ้ำในวันเดียวกัน = ได้รหัสเดิม** (กดรีเฟรช/ขอใหม่ได้ปลอดภัย ไม่เปลืองพูล)

### 2.2 เช็กที่ประตู (ทีมประตูเรียก) — `GET /v2/external/check/<code>/`
```
GET /v2/external/check/1084281961/

Response 200:  { "allow": true,  "member": {...} }     ← เปิดประตู
Response 404:  { "allow": false, "detail": "Code not valid today" }   ← รหัสไม่ได้ใช้วันนี้
Response 403:  { "allow": false, "detail": "Member revoked or not found" }
```
> โดยปกติทีมประตูเป็นคนทำส่วนนี้ ไม่ใช่ reserv — แต่ใส่ไว้ให้เห็นภาพรวม

### 2.3 ที่ api ทำให้หมดแล้ว (reserv ไม่ต้องทำซ้ำ)
- ตรวจ checksum เลขบัตรประชาชน 13 หลัก
- เก็บ/อนุมัติสมาชิก (อนุมัติอัตโนมัติทันที), กันลงทะเบียนซ้ำ
- จัดสรรรหัสจากพูล + หมุนเวียนรายวัน + กันรหัสชนกัน
- บันทึกทุกการเรียกในหน้า Monitor `/monitor/api-usage/` (สืบสวนได้)

## 3. สิ่งที่ reserv ต้องทำ

1. **หน้าเว็บลงทะเบียน/ขอเข้า:** ฟอร์มกรอก เลขบัตร 13 หลัก + ชื่อ + สกุล
2. **เรียก issue จาก backend ของ reserv** (ที่ถือ JWT) — **อย่าเรียกจาก browser ตรงๆ** เพราะ token จะหลุดให้ผู้ใช้เห็น
   - **ใช้ token เดิมของ reserv เอง** (reserv มีบัญชี/token v2 อยู่แล้ว ไม่ต้องสร้าง token ใหม่เฉพาะ external) — ในหน้า Monitor เห็นทั้ง **ระบบ = reserv** (จาก token) และ **endpoint = `ExternalAccessViewSetV2.issue`** → รู้ครบว่าใครทำอะไร
3. **เรนเดอร์ QR** จาก `access_code` ที่ได้ (ใช้ไลบรารี QR ฝั่ง client เช่น qrcode.js ได้เลย — เนื้อใน QR คือสตริง 10 หลักดิบ ไม่ต้องเข้ารหัสเพิ่ม)
4. **บอกผู้ใช้ว่า "QR ใช้ได้เฉพาะวันนี้"** — พรุ่งนี้ต้องขอใหม่
5. **จัดการ error ที่ api ส่งกลับ:** 400 (เลขบัตรผิด → ให้กรอกใหม่), 403 (ถูกระงับ), 503 (พูลเต็ม → แจ้งเจ้าหน้าที่)

## 4. เรื่องที่ต้องเคลียร์/ตัดสินใจ

- [x] **บัญชี/Token:** reserv ใช้ token เดิมของตัวเอง (แต่ละระบบแยกบัญชีตามปกติอยู่แล้ว — ไม่ต้องสร้าง token ใหม่เฉพาะ external) Monitor เห็นทั้ง "ระบบ" (จาก token) + "endpoint"
- [ ] **convention ของประตู:** ประตูจะดู **HTTP status** (200=ผ่าน) หรืออ่าน **field `allow`** ในเนื้อ? ถ้าอยากได้รูปแบบอื่น (เช่น 200 เสมอแล้วดู allow) แจ้งทีม api ปรับ response `check` ให้
- [ ] **ความจุพูล:** ตอนนี้ 100 รหัส/วัน (รับ ≤ ~50 คน/วันสบาย) — ถ้าคาดว่าคนเยอะกว่านี้แจ้งทีม api ขยายพูล
- [ ] **ขอบเขตข้อมูล:** ตอนนี้เก็บแค่ เลขบัตร + ชื่อ-สกุล — ถ้าต้องเก็บเบอร์/อื่นๆ เพิ่ม แจ้งทีม api

---

## 5. ทดสอบ end-to-end (ทำร่วมกันหลัง reserv ต่อเสร็จ)
1. issue เลขบัตรจริง → ได้ `access_code` + `valid_date` = วันนี้
2. check ด้วยรหัสที่ได้ → `allow:true`
3. check ด้วยรหัสมั่ว → `allow:false` (404)
4. รหัสของเมื่อวาน → วันนี้ต้อง `allow:false` (หมดอายุรายวัน)
5. ดูหน้า `/monitor/api-usage/` ว่าทุก call ขึ้น log พร้อมชื่อระบบ

**ติดต่อทีม api:** เมื่อพร้อมเทส หรืออยากปรับสัญญา response แจ้งได้เลย
