## ผลการตรวจ flow ปัจจุบัน

ผู้ใช้ทั่วไปเข้าใช้งานได้ 2 ช่องทาง:

| ช่องทาง | Flow ปัจจุบัน |
|---|---|
| LINE OA | กด Rich Menu → เปิด LIFF → LINE Login → ตรวจการผูกบัญชี → จอง |
| เว็บไซต์ | เปิด `https://lib.npu.ac.th/reserv/` → LINE Login → ตรวจการผูกบัญชี → จอง |

ทั้งสองช่องทางใช้ LINE LIFF เป็นตัวตนหลัก ไม่มี session login แยกสำหรับผู้ใช้ทั่วไปบนเว็บไซต์
Django session login ใช้เฉพาะ Staff Portal (`/manage/`)

การเข้าใช้งานครั้งแรก:
1. ระบบรับ LINE userId จาก `liff.getProfile()`
2. ระบบเรียก `POST /api/check-user/`
3. หากยังไม่ผูกบัญชี ระบบพาไป `/register/`
4. ผู้ใช้เลือกประเภทผู้ใช้ และกรอก LDAP/password
5. ระบบตรวจ LDAP และผูก LINE userId กับบัญชีมหาวิทยาลัย
6. ครั้งถัดไปเข้าใช้งานได้ทันทีโดยไม่ต้องกรอก LDAP ซ้ำ

ประเภทผู้ใช้:
- นักศึกษาเลือก `นักศึกษา`
- อาจารย์และบุคลากรเลือก `บุคลากรภายในมหาวิทยาลัย`

หน้าสาธารณะที่ไม่ต้อง login:
- `/calendar/`
- `/room/<booking_name>/`

## Code Fix 1 — Registration Guard

<!-- Codex edit: low-impact production fix, pending production verification. -->

สถานะ: ✅ ผู้ดูแลระบบทดสอบบน production แล้ว ทำงานถูกต้อง

แก้ `booking/views.py` ให้ตรวจผล `_register_npu_user()` หลัง LDAP ผ่าน:

- หาก NPU API ผูกบัญชีสำเร็จ จึงสร้างหรือ refresh `LineUser` ใน local DB และ redirect ต่อ
- หาก NPU API ผูกบัญชีไม่สำเร็จ ระบบคงอยู่หน้าลงทะเบียนและแสดงข้อความให้ลองใหม่
- ป้องกันข้อมูลค้างที่ local DB เมื่อ source of truth ผูกบัญชีไม่สำเร็จ

ผลกระทบ:
- กระทบเฉพาะผู้ใช้ที่กำลังลงทะเบียนผูกบัญชีใหม่
- ผู้ใช้ที่ผูกบัญชีแล้ว การจอง การยกเลิก Check-in และ IoT ไม่เปลี่ยน behavior

แผนทดสอบ production:
1. ลงทะเบียนผู้ใช้ใหม่ที่ข้อมูล LDAP ถูกต้อง ต้องผูกบัญชีและ redirect เข้าใช้งานได้
2. หาก NPU API ไม่พร้อมใช้งาน ต้องเห็นข้อความ `ไม่สามารถผูกบัญชีกับระบบมหาวิทยาลัยได้ กรุณาลองใหม่อีกครั้ง`
3. ตรวจว่าผู้ใช้เดิมยังเข้าใช้งานและจองได้ตามปกติ

การตรวจสอบก่อน push:
- `AST parse OK: booking/views.py`
- `git diff --check` ผ่าน ไม่พบ whitespace error
- `python manage.py check` ยังรันไม่ได้ใน environment local เพราะยังไม่มี package `python-dotenv` แม้ระบุอยู่ใน `requirements.txt`
- `python -m py_compile booking/views.py` ใช้ไม่ได้ใน sandbox เนื่องจากไม่มีสิทธิ์เขียน `booking/__pycache__` จึงใช้ AST parse แบบไม่เขียนไฟล์แทน

ผลทดสอบ production:
- ผู้ดูแลระบบยืนยันว่าการลงทะเบียนและเข้าใช้งานทำงานถูกต้อง
- พบการหน่วงเล็กน้อยช่วงตรวจสอบสิทธิ์ สันนิษฐานว่าเกี่ยวกับ network latency หรือ external NPU API
- Code Fix 1 ไม่ได้เพิ่มจำนวน external request ในกรณีปกติ เพียงตรวจผล request เดิมก่อนสร้าง local user

## Code Fix 3 — Bind Frontend Cache To LINE User

<!-- Codex edit: low-impact frontend cache isolation fix, verified on production. -->

สถานะ: ✅ ผู้ดูแลระบบทดสอบบน production แล้ว ทำงานถูกต้อง

แก้ profile cache ใน `sessionStorage` จาก key กลาง `npu_user_v1`
เป็น key รูปแบบ `npu_user_v2:<LINE userId>` ในหน้า:

- `booking/templates/booking/landing.html`
- `booking/templates/booking/booking.html`
- `booking/templates/booking/card.html`

แต่ละหน้าจะลบ legacy key `npu_user_v1` เมื่อโหลด profile และอ่านหรือเขียน cache เฉพาะ LINE userId ปัจจุบัน

ผลกระทบ:

- ป้องกัน profile cache ของบัญชี LINE เดิมถูกนำไปแสดงหรือใช้ต่อเมื่อสลับบัญชี
- ผู้ใช้เดิมจะ refresh profile จาก API หนึ่งครั้งหลัง deploy เพราะไม่ใช้ legacy key ต่อ
- ไม่เปลี่ยน backend API, database schema, booking logic, Check-in หรือ IoT

แผนทดสอบ production:

1. ผู้ใช้เดิมเปิดหน้าแรกและหน้าจอง ต้องใช้งานและสร้าง booking ได้ตามปกติ
2. เปิดบัตรสมาชิก ต้องแสดงข้อมูลและ QR Code ของ LINE user ปัจจุบัน
3. สลับบัญชี LINE แล้วเปิดระบบใหม่ ต้องไม่เห็นชื่อ คณะ หรือ QR Code ของบัญชีก่อนหน้า

ผลทดสอบ production:

- หน้าแรกและหน้าจองใช้งานได้ตามปกติ
- บัตรสมาชิกแสดงข้อมูลและ QR Code ถูกต้อง
- สลับบัญชี LINE แล้วไม่พบข้อมูล profile cache ค้างจากบัญชีก่อนหน้า
