## ผลการตรวจ flow ปัจจุบัน

ผู้ใช้ทั่วไปเข้าใช้งานได้ 2 ช่องทาง:

| ช่องทาง | Flow ปัจจุบัน |
|---|---|
| LINE OA | กด Rich Menu → เปิด LIFF → LINE Login → ตรวจการผูกบัญชี → จอง |
| เว็บไซต์ | เปิด `https://lib.npu.ac.th/reserv/` → LINE Login → ตรวจการผูกบัญชี → จอง |

ทั้งสองช่องทางใช้ LINE LIFF เป็นตัวตนหลัก ไม่มี session login แยกสำหรับผู้ใช้ทั่วไปบนเว็บไซต์
Django session login ใช้เฉพาะ Staff Portal (`/manage/`)

การเข้าใช้งานครั้งแรก:
1. ระบบรับ LINE userId จาก `liff.getProfile()`
2. ระบบเรียก `POST /api/check-user/`
3. หากยังไม่ผูกบัญชี ระบบพาไป `/register/`
4. ผู้ใช้เลือกประเภทผู้ใช้ และกรอก LDAP/password
5. ระบบตรวจ LDAP และผูก LINE userId กับบัญชีมหาวิทยาลัย
6. ครั้งถัดไปเข้าใช้งานได้ทันทีโดยไม่ต้องกรอก LDAP ซ้ำ

ประเภทผู้ใช้:
- นักศึกษาเลือก `นักศึกษา`
- อาจารย์และบุคลากรเลือก `บุคลากรภายในมหาวิทยาลัย`

หน้าสาธารณะที่ไม่ต้อง login:
- `/calendar/`
- `/room/<booking_name>/`

## งานค้างที่ควรพิจารณาก่อนประชาสัมพันธ์วงกว้าง

พบจาก static review ยังไม่ได้แก้ code:

1. API สำหรับผู้ใช้หลายจุดรับ `userId` จาก browser โดยยังไม่ได้ตรวจ LIFF access token ฝั่ง server
2. `LineUser.is_active` ยังไม่ถูกตรวจใน `create_booking()`
3. `sessionStorage` ใช้ cache key กลาง `npu_user_v1` ไม่ได้ผูกกับ LINE userId อาจเกิดข้อมูลค้างเมื่อสลับบัญชี

แก้ไขแล้วในรอบนี้:

- แก้ข้อความต้นฉบับโปสเตอร์ให้ระบุว่าเว็บไซต์ยืนยันตัวตนด้วย LINE Login
- อัปเดตเวลาเปิดบริการสำหรับป้ายประชาสัมพันธ์

## ข้อจำกัดเวลาเปิดบริการในระบบจอง

ป้ายประชาสัมพันธ์อัปเดตเวลาเปิดบริการของสำนักวิทยบริการแล้ว แต่ระบบจองยังเก็บ `Room.open_time` และ `Room.close_time`
เป็นเวลาเดียวสำหรับทุกวัน โดย migration ล่าสุดตั้งทุกห้องไว้ `08:30 – 16:30 น.`

ผลกระทบ:
- ฟอร์มจองวันเสาร์ – อาทิตย์ยังแสดง slot ถึง `16:30 น.` เท่านั้น
- backend ยังไม่ได้ตรวจซ้ำว่าเวลาที่ส่งเข้า API อยู่ภายในช่วงเปิดบริการของห้อง

หากต้องการเปิดให้จองวันเสาร์ – อาทิตย์ถึง `17:00 น.` ต้องแก้ code และทดสอบ production แยกจากงาน security

## ลำดับการแก้ code สำหรับ production

เรียงจากผลกระทบน้อยไปมาก เพื่อ deploy และย้อนกลับได้เป็นรอบเล็ก:

| ลำดับ | งาน | ผลกระทบโดยประมาณ | แนวทาง deploy |
|---|---|---|---|
| 1 | ตรวจผล `_register_npu_user()` ก่อนสร้าง local user | ต่ำ | กระทบเฉพาะผู้ใช้ลงทะเบียนใหม่ ทดสอบ register สำเร็จและกรณี NPU API ล้มเหลว |
| 2 | ตรวจ `LineUser.is_active` ก่อนสร้าง booking | ต่ำ | เพิ่ม guard จุดเดียว ผู้ใช้ active เดิมไม่เปลี่ยน behavior |
| 3 | ผูก `sessionStorage` cache key กับ LINE userId | ต่ำ – กลาง | แตะ frontend หน้า landing, booking และ card ผู้ใช้เดิมอาจ login ใหม่หนึ่งครั้ง |
| 4 | บังคับ `LineUser.is_active` ให้ครบทุก API ที่ต้องใช้สิทธิ์ | กลาง | แตะ check-user, my-bookings, cancel, check-in และ IoT control ต้อง regression test flow หลัก |
| 5 | รองรับเวลาแยกวันธรรมดา/วันหยุดสุดสัปดาห์ และตรวจเวลาฝั่ง backend | กลาง | แตะ booking policy, UI slot และหน้า staff schedule ควร deploy แยก commit |
| 6 | ตรวจ LIFF access token ฝั่ง server แทนการเชื่อ `userId` จาก browser | สูง | แตะ contract หลาย API และ frontend หลายหน้า ต้องเตรียมค่า LINE channel ID และทดสอบผ่าน HTTPS บน production |

ทุกลำดับควรทำเป็น commit แยก แล้ว push เพื่อทดสอบ production ทีละรอบ หากพบปัญหาจะย้อนกลับได้ง่าย

## Repository Security Note

ระหว่างตรวจพบว่า Git remote URL เคยฝัง credential ไว้โดยตรง จึงลบ credential ออกจาก local remote config แล้ว
ควร revoke หรือ rotate token เดิมใน GitHub และใช้ Credential Manager หรือ GitHub CLI สำหรับการ push ครั้งถัดไป
