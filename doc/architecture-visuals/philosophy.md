# Controlled Migration

Controlled Migration treats system architecture as a carefully governed movement from fragile dependency toward institutional ownership. Space is used to show authority: external services sit at the edges, the organization-owned core occupies the center, and every arrow carries a visible operational consequence. The work should feel meticulously crafted, with each connection placed through deliberate analysis rather than decorative convenience.

The visual language is structured, quiet, and technical. Forms are rectangular, aligned, and modular, with color acting as an information system: warm warning tones for legacy risk, cool blue for application infrastructure, green for internal data/control, and violet for automation. The composition should look like the product of deep expertise, where every label has been edited down to the minimum needed for precise understanding.

Scale and rhythm express the migration story. The legacy system is narrower and more exposed, with dependencies clustered outward. The new system is broader, layered, and internally coherent, showing that capability has moved closer to the organization. This contrast must be handled with painstaking attention so the result reads as a professional report artifact rather than a rough sketch.

Text is essential-only and integrated into the diagram as visual annotation. Long explanations are avoided; the architecture itself carries the narrative. The final images should appear labored over with care, clean enough for executive presentation, and precise enough for technical review.
