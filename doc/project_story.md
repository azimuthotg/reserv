# การพัฒนาระบบจองพื้นที่บริการ Smart Creative Learning Space
## สำนักวิทยบริการ มหาวิทยาลัยนครพนม

**เวอร์ชันเอกสาร:** 1.0
**วันที่จัดทำ:** เมษายน 2569
**ผู้จัดทำ:** งานเทคนิคสารสนเทศและการจัดการทรัพยากร สำนักวิทยบริการ มหาวิทยาลัยนครพนม

## 1. ที่มาและความสำคัญของการพัฒนาระบบ

สำนักวิทยบริการ มหาวิทยาลัยนครพนม ได้จัดพื้นที่บริการ **Smart Creative Learning Space** เพื่อรองรับการเรียนรู้แบบใหม่ของนักศึกษาและบุคลากร ประกอบด้วยห้องบริการเฉพาะทาง 5 ห้อง ได้แก่ ห้อง Mini Theater, Netflix Room, Canva Room, ChatGPT Room และห้องประชุมชั้น 1 โดยแต่ละห้องมีอุปกรณ์และสิ่งอำนวยความสะดวกที่แตกต่างกันตามวัตถุประสงค์การใช้งาน

เดิมระบบการจองพื้นที่ดังกล่าวพัฒนาขึ้นโดยใช้ **Google Apps Script** ร่วมกับ **Google Sheets** เป็นฐานข้อมูล และเชื่อมต่อกับ **LINE Official Account** ของสำนักวิทยบริการเพื่อให้ผู้ใช้บริการสามารถจองพื้นที่ผ่านแอปพลิเคชัน LINE ได้ ซึ่งเป็นช่องทางที่นักศึกษาและบุคลากรมหาวิทยาลัยนครพนมคุ้นเคยและใช้งานอยู่เป็นประจำ

อย่างไรก็ตาม เมื่อจำนวนผู้ใช้บริการและปริมาณการจองเพิ่มสูงขึ้น ระบบเดิมเริ่มแสดงข้อจำกัดที่ส่งผลต่อประสิทธิภาพการให้บริการอย่างเห็นได้ชัด จึงนำมาสู่การตัดสินใจพัฒนาระบบใหม่บนโครงสร้างพื้นฐานที่มีความมั่นคง ขยายตัวได้ และสามารถบำรุงรักษาได้ในระยะยาว

## 2. ปัญหาและข้อจำกัดของระบบเดิม

### 2.1 ข้อจำกัดด้านประสิทธิภาพ

ระบบ Google Apps Script มีข้อจำกัดด้านการประมวลผลที่กำหนดโดย Google ได้แก่ เวลาในการประมวลผลสูงสุดต่อครั้ง (Execution Time Limit) และโควตาการเรียกใช้งาน API ต่อวัน เมื่อมีผู้ใช้งานพร้อมกันหลายคน ระบบจะทำงานช้าลงหรือเกิดข้อผิดพลาดได้

### 2.2 ปัญหาการจองซ้ำซ้อน (Race Condition)

เนื่องจาก Google Sheets ไม่รองรับกลไก Transaction หรือ Lock ในระดับแถวข้อมูล เมื่อมีผู้ใช้ 2 คนทำการจองช่วงเวลาเดียวกันพร้อมกัน ระบบอาจบันทึกการจองซ้ำซ้อนได้ ส่งผลให้เกิดการขัดแย้งของข้อมูลและสร้างความไม่พึงพอใจแก่ผู้ใช้บริการ

### 2.3 ข้อจำกัดด้านการจัดการข้อมูล

Google Sheets ไม่ใช่ระบบฐานข้อมูลเชิงสัมพันธ์ (Relational Database) จึงขาดความสามารถในการ Query ข้อมูลซับซ้อน การทำ Join ข้อมูลจากหลายตาราง และการตรวจสอบความถูกต้องของข้อมูลในระดับ Schema ส่งผลให้การรายงานและการวิเคราะห์ข้อมูลการจองทำได้ยาก

### 2.4 ความเสี่ยงด้านความต่อเนื่องของบริการ

การพึ่งพาบริการภายนอก (Google Workspace) ทำให้ระบบมีความเสี่ยงหากเกิดการเปลี่ยนแปลงนโยบาย โครงสร้างราคา หรือการหยุดให้บริการจาก Google ซึ่งอยู่นอกเหนือการควบคุมขององค์กร

### 2.5 ข้อจำกัดด้านการพัฒนาต่อยอด

สถาปัตยกรรมของ Google Apps Script ทำให้การเพิ่มฟีเจอร์ใหม่ เช่น การเชื่อมต่อกับอุปกรณ์ IoT เพื่อควบคุมการเปิด-ปิดห้องอัตโนมัติ หรือการออก Virtual Card สำหรับยืนยันตัวตน มีความซับซ้อนและทำได้ยาก

## 3. วัตถุประสงค์ของการพัฒนา

1. พัฒนาระบบจองพื้นที่บริการใหม่บน Web Framework มาตรฐาน (Django) โดยใช้ฐานข้อมูล MySQL เพื่อแก้ไขปัญหาของระบบเดิม
2. รองรับการใช้งานผ่าน LINE LIFF (LINE Front-end Framework) เพื่อให้ผู้ใช้ไม่ต้องเปลี่ยนช่องทางการสื่อสาร
3. ออกแบบสถาปัตยกรรมให้รองรับการต่อยอดในอนาคต ได้แก่ การเชื่อมต่อ IoT และระบบบัตรดิจิทัล
4. Deploy ระบบบนโครงสร้างพื้นฐานของมหาวิทยาลัยที่มีอยู่แล้ว (Windows Server + IIS) โดยไม่กระทบระบบอื่นที่ให้บริการอยู่

## 5. สถาปัตยกรรมและเทคโนโลยีที่ใช้

### 5.1 Tech Stack

| ส่วนประกอบ | เทคโนโลยี | เหตุผลที่เลือก |
|---|---|---|
| Backend Framework | Django 4.2 (Python 3.12) | มาตรฐาน, มี ORM, Admin interface ในตัว, รองรับการพัฒนาต่อยอดได้ดี |
| ฐานข้อมูล | MySQL 8.0 | รองรับ Transaction, Foreign Key, เหมาะกับข้อมูลเชิงสัมพันธ์ |
| Frontend | Bootstrap 5.3 + FullCalendar v6 | Responsive, แสดงปฏิทินการจองแบบ Interactive ได้ |
| LINE Integration | LINE LIFF v2 (SDK 2.15+) | ให้ผู้ใช้จองได้ภายใน LINE app โดยไม่ต้องออกไป browser อื่น |
| Process Manager | NSSM + Waitress | รัน Python WSGI บน Windows Server เป็น Service |
| Reverse Proxy | IIS 10 + Application Request Routing (ARR) | โครงสร้างพื้นฐานที่มีอยู่แล้วของมหาวิทยาลัย |

### 5.2 สถาปัตยกรรมระบบโดยรวม

```
ผู้ใช้กด "จองเลย" ใน LINE OA
        ↓
LINE LIFF เปิด https://lib.npu.ac.th/reserv/booking/?room=X
        ↓
IIS 10 + ARR (Reverse Proxy)
        ↓
Waitress WSGI Server (127.0.0.1:8003)
        ↓
Django Application (C:\project\reserv)
        ↓
    ┌───────────────────┬──────────────────────┐
    │                   │                      │
MySQL 8.0          api.npu.ac.th          LINE Messaging API
(reserv_db)        (LDAP Auth +           (Webhook / LIFF)
                   User Profile)
```

### 5.3 การออกแบบฐานข้อมูล

ระบบประกอบด้วย 4 ตารางหลัก:

| ตาราง | หน้าที่ |
|---|---|
| `Room` | เก็บข้อมูลห้องบริการ, ความจุ, อุปกรณ์, สถานะ |
| `LineUser` | Cache ข้อมูลผู้ใช้ที่ผูก LINE กับ LDAP แล้ว |
| `Booking` | บันทึกการจอง, วันเวลา, สถานะ (confirmed/cancelled) |
| `BookingLog` | Audit trail ทุก action (created/cancelled/accessed/auto_off) |

### 5.4 กลไกป้องกันการจองซ้ำซ้อน

เพื่อแก้ปัญหา Race Condition ของระบบเดิม ระบบใหม่ใช้ `SELECT FOR UPDATE` ของ MySQL ภายใน Database Transaction ทำให้การตรวจสอบและบันทึกการจองเป็น Atomic Operation กล่าวคือ เมื่อผู้ใช้ 2 คนจองช่วงเวลาเดียวกันพร้อมกัน คนที่สองจะได้รับข้อความแจ้งว่าช่วงเวลานั้นถูกจองไปแล้ว แทนที่จะเกิดการบันทึกซ้ำซ้อน

## 6. กระบวนการยืนยันตัวตน (Authentication Flow)

ระบบบูรณาการ 3 ระบบเข้าด้วยกัน ได้แก่ LINE, LDAP ของมหาวิทยาลัย และ api.npu.ac.th

```
เปิดหน้า /booking/?room=X ผ่าน LIFF
        ↓
มี session['line_user_id'] หรือไม่?
    ├── มี → ค้นหา LineUser ใน DB
    │         ├── พบ → แสดงฟอร์มจอง (ข้ามขั้นตอน)
    │         └── ไม่พบ → เรียก api.npu.ac.th เพื่อดึงข้อมูล → แสดงฟอร์มจอง
    └── ไม่มี → redirect ไป /register/ เพื่อผูกบัญชี
                    ↓
            ผู้ใช้กรอก Username + Password (LDAP)
                    ↓
            ระบบตรวจสอบกับ api.npu.ac.th/auth-ldap/
                    ↓
            บันทึก LineUser ลง DB → redirect กลับไปหน้าจอง
```

## 7. แผนการพัฒนา (Phased Development)

โครงการแบ่งการพัฒนาออกเป็น 3 ระยะ เพื่อลดความเสี่ยงและให้สามารถเปิดใช้งานได้เร็วที่สุด

### Phase 1 — ระบบจองหลัก (Core Booking System)
**เป้าหมาย:** แทนที่ระบบเดิมได้ครบถ้วน ผู้ใช้สามารถจองและดูปฏิทินได้ผ่าน LINE
**สถานะ:** ✅ พัฒนาเสร็จและ Deploy แล้ว (เมษายน 2569)

งานที่ดำเนินการ:
- ออกแบบและสร้าง Database Schema (Room, LineUser, Booking, BookingLog)
- พัฒนาระบบยืนยันตัวตนผ่าน LINE LIFF + LDAP
- พัฒนาฟอร์มจองพื้นที่พร้อมตรวจสอบการซ้ำซ้อน
- พัฒนาหน้าปฏิทิน (FullCalendar) แสดงสถานะห้องแบบ Real-time
- ติดตั้งและกำหนดค่าระบบบน Windows Server (NSSM + Waitress + IIS ARR)
- เปิดให้บริการที่ https://lib.npu.ac.th/reserv/

### Phase 2 — IoT Integration (การเชื่อมต่ออุปกรณ์อัจฉริยะ)
**เป้าหมาย:** ควบคุมการเปิด-ปิดอุปกรณ์ในห้อง (Sonoff) อัตโนมัติตามการจอง
**สถานะ:** ⏳ วางแผนไว้ รอดำเนินการหลัง Phase 1 เสถียร

งานที่วางแผน:
- เชื่อมต่อ Home Assistant ผ่าน API (`/api/check-access/`)
- ใช้ `HA_ACCESS_SECRET` (Header: `X-HA-Token`) เพื่อความปลอดภัย
- เพิ่ม field `ha_entity_id` ใน Room model (รองรับไว้แล้วใน Phase 1)
- ทดสอบการเปิด-ปิดอุปกรณ์ตามช่วงเวลาการจองจริง

### Phase 3 — Virtual Card และการบูรณาการกับระบบห้องสมุด
**เป้าหมาย:** ออก Digital Pass สำหรับยืนยันการจอง และเชื่อมกับระบบ Walai
**สถานะ:** ⏳ วางแผนไว้ รอดำเนินการ

งานที่วางแผน:
- ออก Virtual Card พร้อม Barcode (JsBarcode) สำหรับสแกนเข้าห้อง
- เชื่อมต่อระบบ Walai เพื่อตรวจสอบสถานะสมาชิกห้องสมุด
- แสดงสถานะบัตรและประวัติการจองผ่าน LINE LIFF

## 8. กระบวนการ Deploy และการติดตั้งระบบ

### 8.1 สภาพแวดล้อมการติดตั้ง

ระบบถูกติดตั้งบนโครงสร้างพื้นฐานของสำนักวิทยบริการที่มีอยู่แล้ว โดยไม่กระทบระบบอื่นๆ ที่ให้บริการอยู่

| App | ระบบ | Path | Port |
|---|---|---|---|
| App 1 | Project Tracker | `/projects/` | 8000 |
| App 2 | AIMS | `/aims/` | 8001 |
| App 3 | LibLogin | `/liblogin/` | 8002 |
| **App 4** | **Reserv (ระบบนี้)** | **`/reserv/`** | **8003** |

### 8.2 ขั้นตอนการ Deploy หลัก

**Phase A — เตรียม Source Code**
1. ปรับ `settings.py` ให้อ่านค่าทั้งหมดจากไฟล์ `.env` ด้วย `python-dotenv`
2. กำหนดค่า `FORCE_SCRIPT_NAME = '/reserv'` สำหรับ Sub-path deployment
3. เตรียม `deploy/waitress_serve.py` สำหรับรันเป็น Windows Service
4. Push โค้ดขึ้น GitHub Repository

**Phase B — ติดตั้งบน Server**
1. Clone repository ลง `C:\project\reserv`
2. สร้าง Virtual Environment และติดตั้ง dependencies
3. สร้างไฟล์ `.env` พร้อมกรอกค่าจริง (DB credentials, LINE tokens, SECRET_KEY)
4. รัน `migrate`, `load_rooms`, `collectstatic`, `createsuperuser`
5. ติดตั้ง NSSM Service → `SERVICE_RUNNING` บน port 8003

**Phase C — กำหนดค่า IIS**
1. เพิ่ม URL Rewrite Rules ใน `C:\iis_root\web.config` สำหรับ `/reserv/`
2. สร้าง IIS Virtual Directories สำหรับ Static Files และ Media Files
3. Restart IIS และทดสอบทุก App พร้อมกัน

**ผลการทดสอบ:**

```
projects  : 200 OK
aims      : 200 OK
liblogin  : 200 OK
reserv    : 200 OK  ✅
```

## 9. ปัญหาที่พบระหว่างการพัฒนาและการแก้ไข

| ปัญหา | สาเหตุ | วิธีแก้ไข |
|---|---|---|
| `migrate` error: Cannot connect to 'localhost' | ไฟล์ `.env` บน server ไม่มี `DB_HOST` | สร้าง `.env.sample` เป็น template พร้อม `DB_HOST=202.29.55.213` |
| `ModuleNotFoundError: No module named 'django'` | ยังไม่ได้ activate venv หรือ install dependencies | รัน `pip install -r requirements.txt` ใน venv |
| `nssm.exe is not recognized` | PowerShell ต้องระบุ path เต็ม | ใช้ `C:\nssm\nssm.exe` แทน `nssm.exe` |
| ข้อมูลอ่อนไหวถูก push ขึ้น GitHub | `deploy/.env.production` ไม่ได้อยู่ใน `.gitignore` | ลบออกด้วย `git rm --cached` และเพิ่มใน `.gitignore` |
| LIFF ไม่ทำงานบน localhost | LINE LIFF ต้องการ HTTPS เท่านั้น | ทดสอบ LIFF ได้หลัง deploy บน production เท่านั้น |

## 11. สรุป

การพัฒนาระบบจองพื้นที่บริการ Smart Creative Learning Space เป็นการ Migration จากระบบที่พึ่งพาบริการภายนอก (Google Apps Script + Google Sheets) มาสู่ระบบที่องค์กรควบคุมได้เองบนโครงสร้างพื้นฐานของมหาวิทยาลัย โดยใช้ Django Framework และ MySQL เป็นแกนหลัก

ระบบใหม่แก้ไขปัญหาสำคัญของระบบเดิม ได้แก่ ปัญหา Race Condition ในการจองซ้ำซ้อน ข้อจำกัดด้านประสิทธิภาพ และความยากในการต่อยอดพัฒนา พร้อมทั้งออกแบบสถาปัตยกรรมให้รองรับการขยายตัวในอนาคตอย่างเป็นระบบผ่านแผนการพัฒนา 3 ระยะ

ณ เดือนเมษายน 2569 Phase 1 (ระบบจองหลัก) ได้รับการ Deploy และเปิดให้บริการบน `https://lib.npu.ac.th/reserv/` เรียบร้อยแล้ว โดยไม่กระทบการให้บริการของระบบอื่นๆ ที่ดำเนินการอยู่บน Server เดียวกัน

*งานเทคนิคสารสนเทศและการจัดการทรัพยากร สำนักวิทยบริการ มหาวิทยาลัยนครพนม*
