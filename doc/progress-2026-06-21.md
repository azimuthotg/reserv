## สิ่งที่ทำ

### 1. NPU API v2 JWT token layer (ส่วนที่หน้า external ต้องใช้)
reserv เดิมเรียก api.npu.ac.th เฉพาะ endpoint v1 ที่ไม่ต้อง auth จึงยังไม่มี token เลย
เพิ่มชั้นจัดการ token v2 เพื่อให้เรียก `/v2/external/issue/` ได้ (และรองรับ migrate v1→v2 ในอนาคต)

- `reserv/settings.py` — เพิ่ม `NPU_API_USERNAME`, `NPU_API_PASSWORD`, `NPU_API_V2_TOKEN`
- `booking/views.py` — เพิ่ม helper:
  - `_obtain_npu_v2_token()` — POST `/v2/token/` ด้วย username/password → access token
  - `_npu_v2_token(force_refresh)` — cache token ในหน่วยความจำ; ถ้าตั้ง `NPU_API_V2_TOKEN` ใช้ค่านั้นตรง ๆ
  - `_npu_v2_request(method, path, ...)` — แนบ `Authorization: Bearer` + retry 1 ครั้งเมื่อ 401 (token หมดอายุ → ขอใหม่อัตโนมัติจาก username/password)
- `.env.sample` — เพิ่ม section NPU API v2 (แนะนำใช้ username/password เพื่อให้ต่ออายุ token อัตโนมัติ)
- `booking/management/commands/check_npu_v2.py` (ใหม่) — คำสั่งตรวจ token โดยไม่แตะ flow จริง

### 2. หน้า QR บุคคลภายนอก (งาน A จาก handoff)
- `booking/views.py` — view `external_access` (public, ไม่ใช่ LIFF/ไม่ใช่ staff)
  GET = ฟอร์ม, POST → `/v2/external/issue/` ผ่าน token layer → render QR / error
- `booking/urls.py` — route `external/` (prod: `/reserv/external/`)
- `booking/templates/booking/external.html` (ใหม่) — ฟอร์ม ชื่อ-สกุล + เลข 13 หลัก →
  render QR 10 หลัก (qrcodejs ฝั่ง client) + แจ้ง "ใช้ได้เฉพาะวันนี้"
  จัดการ error: 400 (เลขผิด) / 403 (ถูกระงับ) / 503 (pool เต็ม) / เชื่อมต่อไม่ได้
- reserv ไม่เก็บเลขบัตรประชาชน (ส่งผ่านอย่างเดียว — api เป็นผู้ตรวจ checksum/จัดสรรรหัสจาก pool)
- sync ตาราง URL ใน `CLAUDE.md` + `AGENTS.md` เพิ่มแถว `/external/`

## วิธีตรวจสอบ (ต้องทำบน prod — เครื่อง dev รัน Django ไม่ได้)

1. ใส่ `.env` prod: `NPU_API_USERNAME` + `NPU_API_PASSWORD` (เว้น `NPU_API_V2_TOKEN` ว่าง)
2. `nssm restart reserv-booking`
3. `python manage.py check_npu_v2`
   - `✅ obtain token` + `✅ v2 auth check: HTTP 404` = token ใช้ได้
   - `❌ HTTP 401` = สิทธิ์บัญชี reserv ยังไม่เปิด v2 → แจ้งทีม api
4. เปิด `https://lib.npu.ac.th/reserv/external/` กรอกเลขบัตรจริง → เห็น QR + valid_date = วันนี้
5. (ฝั่งประตู/ทีม api) เทสต์รหัสที่ได้กับ `/v2/external/check/<code>/` → `allow:true`

**สถานะการทดสอบ:** ✅ ทดสอบ prod ผ่านแล้ว 2026-06-21
- `check_npu_v2` → `obtain token` สำเร็จ (len=229) + `v2 auth check HTTP 404` (token ผ่าน)
- เปิด `/reserv/external/` กรอกจริง → ได้ QR + รหัส 10 หลัก + valid_date = วันนี้
- Monitor api `/monitor/api-usage/` ยืนยัน: ระบบ=reserv v2, endpoint=ExternalAccessViewSetV2.issue, ผล 200
- เหลือฝั่งประตู integrate `/v2/external/check/<code>/` (มีอยู่แล้วใน api — งานทีมประตู ไม่ใช่ reserv)

## งานค้าง / ตัดสินใจไว้

- **migrate v1→v2 ของ 5 endpoint เดิม** (ldap/register/fetch-user/profile/walai): ยังไม่ทำ — v1 ยังไม่ถูกปิด ไม่รีบ
  - 4/5 ตัว response เข้ากันได้ (แก้แค่ชื่อ path + เติม Bearer): `api→data`, `std-info→student`, `auth-ldap→ldap`, `staff-info→personnel`
  - **walai (1 ตัว) response เปลี่ยนรูป** (array → dict ห่อใน `library_info`) จะพังเงียบ — **ต้องเคลียร์ก่อนปิด v1** (ดู memory `walai-v2-response-incompatible`)
- **ฟีเจอร์ส่งข้อความถึง UID เองใน staff portal:** ตัดสินใจ **ไม่ทำ** — ใช้ LINE OA chat คุย 2 ทางได้อยู่แล้ว
