## บริบท

ระบบ **NMS Agent** (โปรเจกต์แยก ที่ `C:\projects\nms_agent`) เป็น monitoring กลางที่จะเช็กสถานะ
webapp ของสำนักวิทยบริการทุกตัวแบบรายงานทุกเช้า + ถามสถานะผ่าน Telegram ได้ทุกเวลา.
reserv เป็นโปรเจกต์ลำดับแรกที่เพิ่ม health endpoint เพื่อให้ NMS Agent ยิงเข้ามาเช็กได้.

## สิ่งที่ดำเนินการในวันนี้

### เพิ่ม Health endpoint `/health/`

- **`booking/views.py`**
  - เพิ่ม import: `from django.db import connection, transaction` (เดิมมีแค่ `transaction`)
  - เพิ่มฟังก์ชัน `health(request)` — เช็ก DB ด้วย `connection.ensure_connection()` + `SELECT 1`,
    คืน JSON `{status, db, db_ms}` พร้อม HTTP **200** (ปกติ) / **503** (DB มีปัญหา). public ไม่ต้อง login.
- **`booking/urls.py`**
  - เพิ่ม `path('health/', views.health, name='health')` ไว้บนสุดของ urlpatterns
- **`CLAUDE.md` + `AGENTS.md`** — เพิ่มแถว `/health/` → `/reserv/health/` ในตาราง URL Structure (sync ทั้งสองไฟล์)

### รูปแบบ response (มาตรฐานกลางของ NMS Agent)
```json
{ "status": "ok", "db": "ok", "db_ms": 3 }
```
ยึดรูปแบบเดียวกับ saadc2027 ซึ่งมี health endpoint pattern นี้อยู่ก่อนแล้ว.

## วิธีตรวจสอบ
- ✅ `python -m py_compile booking/views.py booking/urls.py` — syntax ผ่าน
- ⚠️ **ยังไม่ได้รัน `manage.py check` / ยิง endpoint จริง** บนเครื่อง dev นี้ (ไม่มี venv/deps `python-dotenv`)
- 🔲 **งานค้าง:** หลัง deploy ขึ้น production แล้ว `nssm restart reserv-booking` และทดสอบ
  `curl https://lib.npu.ac.th/reserv/health/` ควรได้ HTTP 200 + JSON ข้างต้น แล้วบันทึกผลที่นี่

## หมายเหตุ
- endpoint นี้ public โดยตั้งใจ (NMS Agent ยิงจากภายนอกโดยไม่มี session) — เปิดเผยแค่ผลเช็ก DB ไม่มีข้อมูลอ่อนไหว
- ตรวจว่า nginx ส่ง path `/reserv/health/` มาถึง Django (location `/reserv/` ครอบอยู่แล้ว)
